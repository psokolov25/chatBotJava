# Внедрение (детально)

## 1. Цель внедрения
Обеспечить стабильную выдачу талонов через Telegram и доставку событий вызова клиенту.

## 2. Предварительные данные
- Telegram token.
- Перечень филиалов (branch ID, entry point ID, queue system).
- URL, login/password для Orchestra/Axioma.
- Требования по шаблонам уведомлений и клиентскому пути.

## 3. Пошаговое внедрение
1. Подготовить `.env` и проверить `application.yml` placeholders.
2. Заполнить `ORCHESTRA_BRANCHES` и сверить IDs филиалов.
3. Настроить `client_path.yml` (вопросы, переходы, маппинг услуг).
4. Запустить сервис и выполнить smoke-тест.
5. Проверить создание визита в каждой подключённой системе очереди.
6. Проверить контур событий (CometD/Kafka) и доставку уведомления в чат.

## 4. Приёмочные критерии
- В Telegram корректно отображаются филиалы и услуги.
- Создание визита завершается успешно и возвращает талон.
- События `VISIT_CALL`/`VISIT_RECALL` доставляются адресно.
- Ошибки upstream не приводят к падению процесса.

## 5. Риски и как снижать
- Некорректные branch IDs → валидация на этапе pre-prod.
- Ошибки авторизации → отдельный тест credentials до запуска.
- Неустойчивые события → мониторинг Kafka lag/CometD reconnect.

## 6. Go-live checklist
- Секреты вынесены во внешний vault/secret manager.
- Подготовлены runbook L1/L2 и контакты эскалации.
- Настроены логи, метрики и базовые алерты.
