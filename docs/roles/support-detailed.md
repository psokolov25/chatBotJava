# Поддержка (детально)

## 1. Зона ответственности L1/L2
- Восстановление доступности сервиса.
- Первичная диагностика интеграций Telegram/Queue/Kafka/CometD.
- Сбор артефактов для эскалации в разработку.

## 2. Runbook первичной диагностики
1. Проверить доступность:
   - `curl http://localhost:8080/api/bot/status`
2. Проверить runtime-логи:
   - `docker compose logs -f telegram-orchestra-bot`
3. Проверить сценарий пользователя:
   - `/start` → выбор филиала → выбор услуги → получение талона.
4. Проверить upstream доступность:
   - Queue API (Orchestra/Axioma), авторизация, latency.
5. Проверить событийный контур:
   - для Orchestra: CometD подписка,
   - для Axioma: Kafka consumer и lag.

## 3. Карта симптомов
- **401/403 при запросах услуг/визитов**
  - Проверить логин/пароль филиала, корректность base URL, активность учётной записи.
- **Пустой список услуг**
  - Проверить `branchId`, доступность endpoint услуг, соответствие branch в конфигурации.
- **Талон не создаётся**
  - Проверить `entryPointId`, payload параметров и ответ upstream.
- **Нет уведомлений о вызове**
  - Проверить наличие связи `visit ↔ chat`, работу CometD/Kafka и фильтрацию событий.

## 4. Обязательные данные для эскалации
- Точное время инцидента (UTC).
- `branchId`, тип очереди (`orchestra`/`axioma`).
- `chatId`/`customerId` (маскированно при необходимости).
- Фрагменты request/response и статус-коды.
- Stack trace/ошибки из логов.
- Описание ожидаемого и фактического поведения.

## 5. SLA коммуникации
- L1: подтверждение инцидента и первичная классификация.
- L2: диагностика и workaround.
- Dev escalation: полный пакет артефактов из раздела 4.
